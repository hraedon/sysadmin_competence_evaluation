Script A is safe. Script B is not, and the reason is a single parameter difference in a scope field.

Script A defaults SearchBase to "OU=Staff,DC=corp,DC=example,DC=com". Script B defaults SearchBase to "DC=corp,DC=example,DC=com" — the domain root. Running Script B would evaluate every enabled user account in the domain with LastLogonDate older than 90 days, across every OU, and disable all of them. Service accounts, admin accounts, shared mailboxes, resource accounts — anything that passes the filter — would be disabled simultaneously. The rest of the logic is identical; the scope parameter is what distinguishes a controlled Staff OU operation from a domain-wide incident.

Secondary finding present in both: neither script excludes service accounts. Service accounts authenticate as a service principal, not interactively. Their LastLogonDate may be months or years old while the account is actively in use by a running service. Script A's OU scope limits the blast radius of this gap; Script B makes it catastrophic.

The review principle worth naming explicitly: when two scripts look nearly identical, the dangerous differences are almost always in scope parameters, not in the operational logic. The logic can look correct while the scope parameter silently multiplies the impact by orders of magnitude. SearchBase, Filter, Target, Path — these are the parameters that determine what an operation touches. They deserve more scrutiny than the operation itself, especially in time-pressured reviews where scanning for functional correctness can displace reading parameter values.

Pre-run verification step I'd add before using Script A: run Get-ADUser with the same -Filter (-Filter {Enabled -eq $true -and LastLogonDate -lt $cutoffDate}) and -SearchBase ("OU=Staff,DC=corp,DC=example,DC=com") without the Set-ADUser pipe to count and review the accounts that would be affected. Confirm the count is expected before any disable operation runs. One extra step; the kind of thing that prevents incidents.
