The cause is two Active Entra Connect instances writing to the same tenant. The audit log shows Sync_SYNCSRV01_8f4a2c and Sync_SYNCSRV02_3b7e19 alternating writes on J. Reyes's attributes every ~2 hours — each instance overwriting the other's changes on its sync cycle. The Entra Connect Health server list confirms both SYNCSRV01 and SYNCSRV02 are in Active mode. This is not a supported configuration: only one instance should be Active at a time. SYNCSRV02 was deployed as Active when it was set up for the Meridian acquisition three days ago, instead of being placed in Staging mode.

Stopping the conflict requires putting SYNCSRV02 into Staging mode — it will continue syncing from acquired.local locally but stop writing to Entra.

However, the J. Reyes case introduces a complication that matters before taking any action. The background notes confirm she has user objects in both corp.local and acquired.local with different attribute values — her department is Finance in corp.local (SYNCSRV01's source) and Operations in acquired.local (SYNCSRV02's source). These are not two instances racing to write the same value; they are writing genuinely different values from different source objects.

Simply putting SYNCSRV02 in Staging mode will stop the reversions, but it will lock in whichever values SYNCSRV01 wrote last — the corp.local values. If J. Reyes transferred to the acquired entity before the acquisition, the acquired.local values may actually be the correct current record. This needs a deliberate decision about which source is authoritative for her (and potentially other cross-entity employees), not just a "stop the second instance" fix.
