Two Active Entra Connect instances are writing to the same tenant. The audit log shows Sync_SYNCSRV01_8f4a2c and Sync_SYNCSRV02_3b7e19 alternating writes on the same objects every ~2 hours — SYNCSRV01 sets department=Finance, telephoneNumber=+1-555-0198; SYNCSRV02 overwrites with Operations/+1-555-0142, two minutes later. The Entra Connect Health server list confirms both are Active. SYNCSRV02 was deployed as Active when Meridian's IT team set it up for the acquisition, instead of Staging mode.

The immediate fix is to put SYNCSRV02 in Staging mode: it continues syncing from acquired.local locally but stops exporting to Entra. This stops the conflict. But before doing this, check the current state of affected user objects in Entra and determine which instance's values are correct — you want to preserve those values, not accidentally lock in the wrong forest's data when the second instance goes silent.

The J. Reyes case is not just a timing problem. She has objects in both corp.local and acquired.local with different values for the same attributes. SYNCSRV01 and SYNCSRV02 are writing different source data, not racing to write the same value. Putting SYNCSRV02 in Staging mode leaves the corp.local values in place — which may be wrong for a user who transferred to the acquired entity. She (and any other cross-entity employees) needs a deliberate source-of-truth decision and likely a duplicate object merge, not just a sync server change.

The underlying architecture problem is also not resolved by Staging SYNCSRV02. The correct design for merging two AD forests into one Entra tenant is a single Entra Connect instance configured with multiple forest connectors — one connector to corp.local, one to acquired.local — with explicit attribute precedence rules defining which forest wins for objects that exist in both. That design allows both forests to sync while giving the admin control over conflict resolution. Two separate Active instances cannot provide this because each instance applies its own full export to the tenant without knowledge of what the other is writing.

The remediation path: (1) Determine which server's current values in Entra are correct for each affected user, preserving that state. (2) Put SYNCSRV02 in Staging mode to stop the conflict. (3) Resolve the J. Reyes duplicate object — decide which forest is authoritative, use Entra Connect's soft-match or hard-match to merge the objects, and update the non-authoritative forest to match. (4) Plan the proper multi-forest connector configuration on SYNCSRV01 (or a new dedicated sync server), with attribute precedence rules, before putting SYNCSRV02 back into any Active role.
