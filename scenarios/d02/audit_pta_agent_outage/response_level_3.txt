The authentication failures are caused by both PTA agents being on the same server, SYNCSRV01, which is now offline for emergency patching.

The architectural mistake: both pta-agent-001 and pta-agent-002 are registered on SYNCSRV01 (192.168.10.45). Their heartbeats died within 3 seconds of each other at 02:14 UTC, right before SYNCSRV01 went offline at 02:15. Incident ticket INC-2025-0917 closed in September with "Redundancy requirement met" — but co-locating both agents on one host means the redundancy only covers agent process failures, not host failures. This is the failure mode that wasn't accounted for.

The PTA availability model is what makes this so impactful: Pass-Through Authentication routes every fresh authentication request to an on-premises agent. With no agents available, there is no cloud-side fallback — Entra cannot authenticate users independently. The sign-in log error is explicit: "No Pass-Through Authentication Agents are available." This is the core tradeoff between PTA and Password Hash Sync. PHS stores a derived password hash in Entra ID and can authenticate users even during a complete on-premises outage. PTA's appeal is that credentials never leave the on-premises environment, but the price is this on-prem dependency for every fresh authentication.

It's worth noting the blast radius is not "all users offline." The sign-in log shows some successful sign-ins (agarcia at 02:21, jsmith at 03:28) labeled "session refresh — no auth." Users with valid existing sessions — access tokens under an hour old, or persistent refresh tokens — can continue working without triggering a new PTA authentication. The outage affects new sign-in attempts and applications that require fresh authentication.

The prevention is straightforward: PTA agents must be on separate physical hosts. Switching to PHS would eliminate the agent dependency entirely, but that's a deliberate architecture change requiring a change process — not something to do during an active incident.
