The root cause is missing SPNs. The setspn -L output for corp\svc_sql shows no entries; both setspn -Q queries for MSSQLSvc/SQLSRV01.corp.local and MSSQLSvc/SQLSRV01 return "No such SPN found." The SQL Server error log from 15:48 on 2026-03-20 documents this explicitly: SPN registration for MSSQLSvc/SQLSRV01.corp.local:1433 failed with 0x2098, meaning corp\svc_sql lacks Write SPN permission in Active Directory. The service started successfully — the SPN error is a warning at startup, not a service failure, which is why the BI team saw "service started successfully" and closed the change.

FQDN connections (SQLSRV01.corp.local) trigger Kerberos. Kerberos requires the client to request a service ticket from the KDC by looking up the SPN MSSQLSvc/SQLSRV01.corp.local:1433. With no such SPN registered, the KDC has nothing to issue a ticket against, and the connection fails with "Cannot generate SSPI context." IP address connections use NTLM instead — NTLM is a challenge-response protocol that doesn't require an SPN — which is why K. Strand's connection to 192.168.10.20,1433 works fine.

Why the BI team didn't detect this: the change record shows they tested using hostname SQLSRV01 from a local connection. Short-name connections are more likely to fall back to NTLM than FQDN connections, and a local connection may bypass Kerberos entirely. The FQDN path — which is the production connection string used by all external applications — was never tested before the change was closed.

The fix: manually register the SPNs using setspn with an account that has AD write rights. The service account corp\svc_sql cannot auto-register (0x2098 error confirms this), so the registration must be done by an account with the necessary permissions — typically a Domain Admin running setspn on a domain-joined machine.
