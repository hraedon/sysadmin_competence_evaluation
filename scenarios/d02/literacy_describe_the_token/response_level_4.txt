(1) Ticket #1 is a CIFS service ticket for FILESERVER01. It authenticates jsmith's identity — it is a credential presentation mechanism, not an authorization grant. What jsmith can actually do on FILESERVER01 (which shares are accessible, which files can be read or written) is determined entirely by ACLs on the server. The ticket is the mechanism by which FILESERVER01 knows who it's talking to; the permissions are a separate decision made by the server after that recognition.

(2) The KDC is LABDC01.corp.example.com, visible in the "KDC Called" field. LABDC01 is the domain controller acting as the Kerberos Key Distribution Center for the CORP.EXAMPLE.COM realm. Both tickets were issued by the same KDC.

(3) The service ticket expires at 18:03:22. Loss of access is not immediate. The OS will silently request a new CIFS service ticket from LABDC01 on the next access attempt, provided the TGT is still valid. If the TGT has also expired, re-authentication may be required. An already-established SMB session (open file handles, connected drive) may persist past ticket expiry because authentication occurred at session establishment — the session doesn't re-authenticate per request. A new connection after expiry would require a fresh ticket.

(4) NTLM fallback: accessing FILESERVER01 by IP address instead of FQDN (no SPN resolution); CIFS/FILESERVER01 SPN absent from AD; client not domain-joined; port 88 blocked to LABDC01; target outside the realm with no configured trust.

One additional observation worth noting: the CIFS ticket carries the ok-as-delegate flag, which is absent on the LDAP ticket. This flag indicates FILESERVER01 is trusted for Kerberos constrained delegation — it can request service tickets on jsmith's behalf to other services downstream. This is relevant in scenarios where FILESERVER01 acts as a front-end: a DFS namespace that redirects to a backend share, a file server integrated with a SQL backend, any multi-tier architecture where FILESERVER01 needs to make requests to another service while impersonating the connected user. The absence of ok-as-delegate on the LDAP ticket means LABDC01 is not trusted for the same kind of delegation, which is the expected and correct configuration for a domain controller.
