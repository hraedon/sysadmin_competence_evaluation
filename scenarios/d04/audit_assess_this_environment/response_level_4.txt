Four risks, triaged correctly.

Immediate: CRL expires in 16 days. Publish a new CRL now. When the CRL expires, revocation checking fails — hard-fail clients stop trusting the entire PKI; soft-fail clients silently stop enforcing revocation. 16 days is an operational deadline, not a planning horizon. The answer is not OCSP deployment (medium-term improvement), not redesigning revocation infrastructure, not investigating why the CRL hasn't been renewed. The answer is: publish a new CRL from the issuing CA this week. Then investigate why it wasn't renewed on schedule.

High priority structural: root CA is online. The root CA is a VM on the production cluster with continuous network exposure. The root CA should be offline — powered off and network-disconnected — except during rare intermediate CA signing events. An online root CA is continuously attackable. If compromised, the attacker can issue trusted certificates for any domain; every client in the organization would trust those certificates. The blast radius of root CA compromise is total PKI replacement: re-issue every certificate in the hierarchy, re-provision trust anchors on every client, and rebuild client trust in the PKI. This is the highest-consequence risk in the environment even though it is not the most immediate. Plan an offline root CA migration with a defined timeline.

Moderate: tracking spreadsheet is 8 months stale. This is what caused the load balancer incident. A stale tracking process means you lose visibility into what's expiring. This is a process gap, not an immediate operational risk — certificates don't expire faster because of a stale spreadsheet. Address after the CRL and root CA issues with automated monitoring tooling.

Moderate (architectural risk): autoenrollment template scopes to all domain computers. Currently this is not a problem, but any future change to this template needs careful testing before deployment — a template misconfiguration affecting all domain computers simultaneously is a large blast radius. This is a change risk to document, not an immediate remediation item.

Additional finding: code signing "with whatever certificate is available" is a provenance gap. Code signing certificates should come from a controlled template with defined key usage. Uncontrolled code signing means you can't audit or revoke code signing certificates systematically, and you may have certificates expiring without tracking. Lower priority than the CRL and root CA, but worth surfacing as an unaddressed risk.
