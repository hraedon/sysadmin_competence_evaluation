Two critical findings. I will not approve this request as submitted.

Critical finding 1: trust scope is unconditional. Adding VendorConnect's CA to the Windows trust store grants that CA trust for any domain in any certificate it issues — not just VendorConnect's application, not just vendorconnect.com. If VendorConnect's CA issues a certificate for "login.microsoftonline.com" or "corp.example.com," every computer receiving this GPO will trust it. This is an organization-wide trust delegation to an external party's CA. The request describes this as an application integration. It is a trust boundary decision with organization-wide security implications.

Critical finding 2: CRL distribution point is unreachable. The CDP is "http://pki.vendorconnect.internal/crl" — an internal VendorConnect network path our computers cannot access. Revocation checking will fail for every certificate issued by this CA. Soft-fail configuration means clients silently continue trusting (revocation checking is effectively disabled). Hard-fail means connections fail entirely. In the compromise scenario, our clients have no mechanism to learn that a VendorConnect certificate has been revoked.

Governance questions required before I could reconsider:
- Who holds the private key for this CA, how is it protected (HSM vs software), and what is VendorConnect's key management policy?
- What is their certificate issuance policy — what checks do they run before issuing a certificate, and for what domains/organizations?
- What is the contractual obligation if VendorConnect's CA is compromised and we need to remove the trust?
- Is there a CRL/OCSP endpoint accessible from our network? If not, this is a hard no.

Conditions for approval if the governance questions are answered satisfactorily:
1. CRL or OCSP endpoint accessible from our domain computers — not negotiable.
2. Deployment scoped to the machines that actually run VendorConnect, not all domain computers. GPO to all computers is overkill for a single-vendor application and maximizes blast radius unnecessarily.
3. A trust removal runbook documented before deployment — how we remove this CA from the trust store if the vendor relationship ends or the CA is compromised.
4. Pilot deployment with testing before organization-wide rollout.

On the two-week timeline: this request requires governance review that cannot be completed in two weeks responsibly. The timeline should be extended or the vendor needs to provide acceptable answers quickly. "We need this in two weeks" is not a security argument.
