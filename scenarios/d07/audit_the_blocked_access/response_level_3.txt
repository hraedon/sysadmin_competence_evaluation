The junior admin's investigation missed the subnet-level NSG because of a gap in how Azure NSG evaluation works. In Azure, NSGs at the subnet level and NIC level are evaluated independently, and for inbound traffic the subnet NSG is evaluated first. If the subnet NSG drops a packet, it is discarded before the NIC NSG is ever consulted. Checking the NIC NSG and finding an allow rule does not confirm the traffic will pass — it only confirms that if the traffic reaches that layer, it will be allowed.

The junior admin checked nsg-portal-nic and found Allow-HTTPS-Any at priority 100. That rule is correct and would allow the traffic — but it is never reached for external connections, because the subnet NSG drops them first. Reading nsg-webapp-subnet: priority 100 allows TCP 443 from 10.0.0.0/8 (corporate and VPN addresses), priority 200 allows TCP 80 from Any, and the default at 65500 is DenyAllInbound. There is no rule allowing TCP 443 from internet source addresses. External partner traffic on port 443 is dropped at priority 65500 without the NIC NSG being consulted at all. Corporate and VPN traffic works because those source IPs fall within 10.0.0.0/8, matching the priority 100 subnet NSG allow.

The fix is to add an inbound allow rule to nsg-webapp-subnet for TCP 443 from Internet, at a priority between the existing allow rules and the default deny. Adding the rule to nsg-portal-nic is not necessary — it already has the correct allow rule in place.
