The junior admin's investigation missed the subnet-level NSG because of a gap in how Azure NSG evaluation works. In Azure, NSGs at the subnet level and NIC level are evaluated independently, and for inbound traffic the subnet NSG is evaluated first. If the subnet NSG drops a packet, it is discarded before the NIC NSG is ever consulted. Checking the NIC NSG and finding an allow rule does not confirm the traffic will pass — it only confirms that if the traffic reaches that layer, it will be allowed.

The junior admin checked nsg-portal-nic and found Allow-HTTPS-Any at priority 100. That rule is correct and would allow the traffic — but it is never reached for external connections, because the subnet NSG drops them first. Reading nsg-webapp-subnet: priority 100 allows TCP 443 from 10.0.0.0/8 (corporate and VPN addresses), priority 200 allows TCP 80 from Any, and the default at 65500 is DenyAllInbound. There is no rule allowing TCP 443 from internet source addresses. External partner traffic on port 443 is dropped at the subnet NSG's default deny, without the NIC NSG being consulted at all. Corporate and VPN traffic works because those source IPs fall within 10.0.0.0/8, matching the priority 100 subnet NSG allow.

The immediate fix is to add an inbound allow rule to nsg-webapp-subnet for TCP 443 from Internet at a priority below the existing allow rules. The NIC NSG already has the correct rule.

Two secondary concerns worth noting. First, adding a TCP 443 allow to nsg-webapp-subnet applies to every VM in subs-webapp (10.0.1.0/24), not just portal-prod-01. If other VMs in the subnet are not intended to be publicly reachable on 443, the subnet rule is broader than needed. In that case the correct pattern is to make the subnet NSG permissive for internet HTTPS and rely on the NIC NSG for per-VM control — which is already configured correctly on nsg-portal-nic. The cleanest long-term state is: permissive subnet NSG for traffic types that are controlled per-VM, with restrictive NIC NSGs on each VM. Mixing restrictive rules at both layers produces exactly this investigation confusion.

Second, the efficient first diagnostic step for this class of problem is Azure Network Watcher's IP Flow Verify: given a source IP, destination IP, port, and direction, it identifies which rule is controlling the traffic and which NSG layer contains it. That would have pointed directly to the subnet NSG rule within seconds, rather than requiring manual review of each layer in sequence.
