The root cause is VM family selection. The Standard_B2ms is a burstable VM. Azure B-series VMs operate on a CPU credit model: credits accumulate when CPU utilisation is below the VM's baseline percentage and are spent when CPU exceeds the baseline. For the B2ms, the baseline is 40% of 2 vCPUs. When the credit balance reaches zero, the VM is throttled to that baseline and cannot exceed it regardless of demand.

The metrics show this precisely. At go-live, the VM started with a full credit balance (60 credits) and the application ran at 78–84% CPU for the first 75 minutes — drawing down credits while processing at full capability. At 11:15 UTC the credits reached zero. From 11:30 onward, CPU is pinned at 39–41% — exactly the B2ms sustained baseline. The VM is not struggling or failing. It is operating correctly within its throttled ceiling. The service is processing as fast as the VM will allow in its depleted state.

The three-day test period did not surface this because B-series VMs start with a full credit balance. If the test load did not sustain CPU above 40% consistently, the credits never depleted, and the VM appeared capable of higher CPU throughput throughout testing. Production load was the first workload to continuously exceed the baseline, and the credit exhaustion happened within the first two hours. The test results are not valid evidence of production suitability for a sustained-CPU workload.

The fix is to move to a non-burstable VM family. A Standard_D2s_v3 (2 vCPU, 8 GiB RAM) provides the same memory with full sustained access to both vCPUs without a credit model. Disk, network, and memory are not constrained on the current VM — only CPU — so an in-place resize to D2s_v3 addresses the issue.
