The root cause is VM family selection. The Standard_B2ms is a burstable VM. Azure B-series VMs operate on a CPU credit model: credits accumulate when CPU utilisation is below the VM's baseline percentage and are spent when CPU exceeds the baseline. For the B2ms, the baseline is 40% of 2 vCPUs. When the credit balance reaches zero, the VM is throttled to that baseline and cannot exceed it regardless of demand.

The metrics show this precisely. At go-live, the VM started with a full credit balance (60 credits) and the application ran at 78–84% CPU for the first 75 minutes — drawing down credits while processing at full capability. At 11:15 UTC the credits reached zero. From 11:30 onward, CPU is pinned at 39–41% — exactly the B2ms sustained baseline. The VM is not struggling. It is operating correctly within its throttled ceiling, processing as fast as the VM will allow.

The three-day test period did not surface this because B-series VMs start with a full credit balance. If the test load did not sustain CPU above the 40% baseline, credits never depleted and the VM appeared fully capable. Production sustained load is the first workload that continuously exceeds the baseline, and exhaustion happened within two hours. A valid performance test for a sustained processing workload requires running representative throughput long enough to exhaust the initial credit balance and confirm performance at the throttled baseline — which is the production performance floor, not a peak.

The fix is to move to a non-burstable VM family. A Standard_D2s_v3 (2 vCPU, 8 GiB RAM) provides the same memory with full sustained access to both vCPUs. Disk, network, and memory are not constrained on the current VM, so an in-place resize resolves the issue.

The deeper problem is the workload classification. B-series VMs are designed for workloads that are idle most of the time with occasional CPU spikes: development environments, low-traffic web servers, intermittent jobs. The ProcessingWorker is the opposite — a continuous pipeline that runs at full throughput as long as the queue has messages. Any B-series VM will reach the same state under this workload: a B4ms has a 40% baseline of 4 vCPUs, which delays exhaustion but does not prevent it. The correct selection framework is: idle or bursty workloads → B-series; sustained CPU workloads → D/F/compute-optimised series. The cost comparison in the provisioning notes was invalid because it assumed equivalent sustained CPU capacity, which B-series cannot provide for this workload class — the "cheaper" VM delivered less throughput per dollar than the D2s_v3 would have.
