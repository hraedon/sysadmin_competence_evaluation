This job is not meeting its retention target. The job is configured to keep 7 restore points, but only 3 are available: the full from 2026-03-16 and incrementals from 2026-03-17 and 2026-03-18. There are no restore points from 2026-03-19 through 2026-03-22. Any recovery request for those four days cannot be fulfilled.

The session log explains the mechanism. Veeam's forever-forward incremental mode automatically merges the oldest restore point into the full backup when repository free space drops below the threshold. Each night for the last four nights: (1) the incremental backup ran and completed — new data transferred from source; (2) free space dropped below 5%, triggering a merge — oldest restore point deleted. Both operations succeeded, so the job reports Success. Net restore point count: unchanged. The "data transferred" column is real and accurate; the backup data from each night is present in the updated full backup file. But there is no separate restore point for 2026-03-19, 2026-03-20, 2026-03-21, or 2026-03-22.

The root cause is that the repository cannot hold the configured retention target. Three restore points consume 3.92 TB of a 4 TB repository, leaving insufficient space for a fourth without a merge. The retention policy and repository size are inconsistent, and the inconsistency manifests as silent merge cycles rather than a job failure.

The fix is to expand the repository. The monitoring configuration also needs to be updated. The current alerting on job status — green means healthy — cannot detect this failure mode because the job succeeds. Backup health requires monitoring three additional signals: restore point count (alert when it falls below the retention target), repository free space (alert before the merge threshold is reached, not after), and RPO age (alert when the oldest restore point is older than the target recovery window). Seven consecutive green lights in the job history are accurate but are not sufficient evidence that this job is healthy. Monitoring that only checks job status creates a false confidence signal in exactly the failure mode this job is currently in.
