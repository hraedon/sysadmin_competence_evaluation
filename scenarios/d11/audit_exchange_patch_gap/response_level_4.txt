The server is six Exchange CUs behind, and the SCCM report cannot show this because SCCM Software Updates only manages WSUS-distributed patches. Exchange CUs are standalone installers. The report's silence on Exchange is not a finding of compliance — it is a coverage gap.

The SCCM report lists 47 Windows updates, all installed, 100% compliant. Exchange is absent entirely — not non-compliant, not errored, simply not tracked. The artifact's own note makes this explicit: "SCCM Software Updates manages only patches distributed through WSUS." Exchange CU9 (15.2.986.14) was installed on 2025-09-14 — the last month the previous administrator was present. CU15 (15.2.1544.4) is current as of 2026-02-11. Six CUs spanning approximately 18 months of releases.

The Exchange Health Manager "Healthy" status reflects operational state: services are running, the database is mounted, the certificate is valid. These checks are correct. They do not assess software currency. "Healthy" in this context means "functioning as configured," not "current." Both the SCCM report and the Health Manager are telling the truth about what they measure — the problem is that neither measures Exchange patch state, and the combination creates a plausible appearance of a fully-managed server.

The delivery trend is the confirming evidence. Queue depth grew from 12 (September) to 5,103 (today); NDRs grew from 2/day to 1,402/day; delivery success dropped from 99.8% to 81.2%. The NDR reason is TLS negotiation failure. The trajectory is gradual and continuous, starting from September 2025 — consistent with external mail systems progressively updating their minimum TLS requirements in ways that CU9's transport layer cannot satisfy, while CU10 through CU15 contain the relevant fixes.

Remediation path: Exchange CUs must be deployed outside the SCCM Software Updates pipeline. Options are a manually staged deployment following the Exchange CU installation procedure, or an SCCM Application Deployment package built around the Exchange Setup.exe installer. Either way, this needs to happen as a critical remediation — six CUs behind on an internet-facing mail server is not a routine catch-up task, and the delivery failure rate (now at 18.8%) is actively impacting business operations.

Process fix: the SCCM Software Updates compliance report scope should be documented explicitly. Any administrator who looks at this report and sees "Compliant" should understand what that means and what it excludes. The correct process for Exchange patch tracking needs to be established and owned, not assumed to be covered by the Windows patching pipeline.
