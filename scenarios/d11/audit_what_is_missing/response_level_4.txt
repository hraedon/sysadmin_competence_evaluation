The two compounding failures:

FAILURE 1: Wrong audit category — audit policy blinds FILESRV01 to file share access

The audit policy output (attached to the investigation notes) shows "Audit File Share: No Auditing" and "Audit File System: No Auditing" under the Object Access category. File share access events are generated under Object Access, not Logon/Logoff. The events the junior administrator searched for — 4624 (logon success) and 4625 (logon failure) — are generated when a new logon session is established, not when a user accesses a specific share or file within an existing session.

To understand the distinction: when a domain user maps a drive or accesses a UNC path on a domain file server, two things happen:
1. A Kerberos service ticket is requested from the DC for CIFS/FILESRV01
2. The SMB connection is established and the share is accessed

If the connection creates a new logon session on FILESRV01 (Logon Type 3, Network), Event 4624 is generated on FILESRV01 — if and only if the Logon audit category is recording successes, which it is. However, this logon event represents the session creation, not share access. Subsequent file operations within the same session generate no further logon events. More importantly, the specific share accessed, the files opened, and the operations performed are captured only by Object Access events (5140 for share access, 4663 for object access within the share) — and only if Object Access auditing is enabled AND the share/files have SACL entries configured.

FILESRV01 has no Object Access auditing enabled. The Security log is categorically blind to share and file access. The junior administrator searched the right log for logon events and found nothing — but found nothing is not the same as nothing happened, because the log would not have recorded what happened even if it did.

FAILURE 2: Log retention does not cover the incident window

Security log earliest entry: 2026-03-16. Incident window: 2026-03-02. The log rotates on a 20MB limit with "overwrite as needed." On an active file server that generates logon events for every user session, 20MB is exhausted in days to weeks depending on load. Any entries from 2026-03-02 were overwritten before 2026-03-16.

This compounds Failure 1: even if logon events were the right evidence, they are gone. The current absence of entries from 2026-03-02 is a log rotation artifact, not evidence of absence of activity.
