The script's deployment steps are correct. The problem is in the diagnosis and the fix.

The comment is the tell. "Checked ownership and permissions multiple times — they are correct" is a complete diagnostic by the Windows permission model. On a RHEL 9 system with SELinux enforcing, it is an incomplete one. File access is controlled by two independent layers: DAC (the owner and mode that chown and chmod set) and MAC (the SELinux context label on the file). Correct DAC permissions do not guarantee access if the MAC context is wrong. The 403 was a MAC denial. The diagnostic tools are different: ls -Z to see the context on the files, and ausearch -m AVC -c httpd or journalctl -t setroubleshoot to read the specific denial from the audit log. The admin never checked that layer.

setenforce 0 does not fix a context problem — it removes SELinux enforcement for the entire host. Every process on this system that was previously confined by policy is now unconfined. The admin applied a system-wide security regression to solve what is a labeling problem in one directory. The fix also does not survive a reboot: setenforce only changes the runtime mode. After the next reboot, the system returns to enforcing and the 403 comes back.

The correct fix is to label /opt/crm with the type Apache's policy allows it to read. For httpd serving content, that type is httpd_sys_content_t. The fix belongs in the deploy script: chcon -R -t httpd_sys_content_t /opt/crm, or semanage fcontext to create a persistent policy entry followed by restorecon to apply it. Either approach is surgical — it gives Apache the access it needs without touching the policy for any other process on the system.
