The network team's DNS cleanup change is the cause. The pre-cleanup configuration included a conditional forwarder routing blob.core.windows.net queries to 168.63.129.16 (Azure DNS). That forwarder was removed. Without it, on-premises clients resolve blob.core.windows.net through public DNS and receive the storage account's public IP (20.150.47.130). The storage account has public access disabled, so the public IP returns a connection refused.

The DNS comparison in the artifact confirms this: on-premises resolves to 20.150.47.130 (public), while the Azure VM resolves to 10.0.2.5. Restoring the conditional forwarders for blob.core.windows.net and privatelink.blob.core.windows.net pointing to 168.63.129.16 on the on-premises DNS servers will fix the issue. No changes to the Azure configuration are needed.
