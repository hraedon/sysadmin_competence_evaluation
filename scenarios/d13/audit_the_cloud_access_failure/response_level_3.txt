The network team's DNS cleanup change is the cause. The pre-cleanup on-premises DNS configuration included conditional forwarders routing blob.core.windows.net and privatelink.blob.core.windows.net queries to 168.63.129.16 (Azure DNS). Those forwarders were removed. Without them, on-premises clients resolve blob.core.windows.net using public DNS and receive the storage account's public IP (20.150.47.130). The storage account has public access disabled, so the public IP returns a connection refused.

The DNS comparison in the artifact makes the resolution path difference explicit: on-premises gets 20.150.47.130 (public); the Azure VM gets 10.0.2.5 (private endpoint IP). The Azure VM uses Azure's internal DNS resolver, which is aware of the private DNS zone (privatelink.blob.core.windows.net) linked to the VNet and returns the private endpoint IP directly. On-premises clients cannot use Azure's internal DNS resolver without a forwarder routing their queries there first.

The resolution chain for private endpoints from on-premises requires three components working together: the private endpoint and its private DNS zone (both present and unchanged in Azure), a VNet link between the private DNS zone and the VNet (unchanged), and a conditional forwarder on the on-premises DNS servers routing the relevant hostname to Azure DNS (removed during cleanup). The third component is the one that was removed.

The fix is to restore the conditional forwarders for blob.core.windows.net and privatelink.blob.core.windows.net pointing to 168.63.129.16. No Azure changes are required.
