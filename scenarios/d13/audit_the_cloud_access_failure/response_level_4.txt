The network team's DNS cleanup change is the cause. The pre-cleanup on-premises DNS configuration included conditional forwarders routing blob.core.windows.net and privatelink.blob.core.windows.net queries to 168.63.129.16 (Azure DNS). Those forwarders were removed. Without them, on-premises clients resolve blob.core.windows.net using public DNS and receive the storage account's public IP (20.150.47.130). The storage account has public access disabled, so the public IP returns a connection refused.

The DNS comparison confirms the resolution path difference: on-premises gets 20.150.47.130 (public); the Azure VM gets 10.0.2.5 (private endpoint IP). Azure VMs use Azure's internal DNS resolver, which returns the private endpoint IP because the private DNS zone (privatelink.blob.core.windows.net) is linked to the VNet. On-premises clients cannot reach Azure's internal DNS infrastructure without a forwarder routing queries there. That forwarder was the removed entry.

The private endpoint DNS resolution chain from on-premises has three required components: the private endpoint and private DNS zone in Azure (present and unchanged), the VNet link between the DNS zone and the VNet (present and unchanged), and a conditional forwarder on the on-premises DNS servers sending the relevant hostname queries to 168.63.129.16 (removed). The network team's statement — "we didn't touch anything in Azure" — is accurate. They touched the on-premises prerequisite.

The fix is to restore the conditional forwarders. No Azure changes are required.

The change management failure here is a mental model gap, not a process failure in the narrow sense. The network team's DNS housekeeping project operated under a reasonable assumption — that on-premises DNS forwarders and Azure DNS are separate administrative domains. For most forwarder rules, that's true. For forwarders routing to 168.63.129.16, it's not: those rules are the on-premises half of a hybrid connectivity chain, and removing them is operationally equivalent to removing a network route to a cloud service. The pre-change review for any DNS housekeeping project should identify forwarders routing to Azure DNS (168.63.129.16) as a distinct category requiring impact assessment before removal.
